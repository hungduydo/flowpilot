5. **Speed difference is significant** — Anthropic models are ~3-4x faster than Ollama Cloud

## Scoring Formula

```
overall_score = valid_rate × 25 + (1 - fix_rate) × 25 + node_accuracy × 25 + param_accuracy × 25
```

| Component | Weight | Description |
|-----------|--------|-------------|
| Valid rate | 25 pts | % of workflows passing validation |
| Fix rate | 25 pts | Lower auto-fix count = higher score |
| Node accuracy | 25 pts | % of expected node types found in output |
| Param accuracy | 25 pts | % of expected parameters correct |

---

## Metric Definitions

| Metric | Meaning |
|--------|---------|
| **Valid workflows** | % of generated workflows that pass the 3-layer validator (schema + node types + graph) |
| **Avg fixes** | Average number of auto-corrections applied by `_post_process()` per workflow. Lower = the LLM understood n8n format better. Examples: Slack `webclient→message`, Schedule cron string→interval object, If conditions string→nested object |
| **Node accuracy** | % of expected node types present in the generated workflow (e.g., prompt asks for Slack+Webhook, both found = 100%) |
| **Param accuracy** | % of expected parameters correct before post-processing (e.g., Slack `resource: "message"`, `operation: "send"`) |
| **Avg time** | Average wall-clock time per workflow generation (includes plan phase + JSON generation + retries) |
| **Delta** | Score difference between `all` layers and `none` (positive = layers help) |

## Intelligence Layer Impact Analysis

### When Do Layers Help?

| Scenario | Impact | Evidence |
|----------|--------|----------|
| **Anthropic models + param accuracy** | Strong positive (+10%) | Both Haiku versions show consistent improvement |
| **Ollama qwen3.5 overall** | Neutral | Scores nearly identical with/without layers |
| **Auto-fix reduction** | Minimal | Fix counts don't change much — post-processor handles these regardless |
| **Generation speed** | Slight negative | Extra context tokens add ~1-3s to generation time |

### Why Qwen Shows Less Improvement

1. **Already trained on n8n patterns** — qwen3.5:397b may have more n8n-related training data
2. **Larger model size** — 397B parameters vs Haiku's smaller size, so it "knows more" without context
3. **Different failure modes** — qwen3.5 fails on node selection (githubTrigger), not parameters, and RAG context doesn't help choose correct trigger nodes

### Recommended Configuration

| Use Case | Recommended Model | Layers |
|----------|------------------|--------|
| **Production (quality priority)** | `anthropic:claude-haiku-4-5-20251001` | `all` |
| **Production (cost-effective)** | `ollama:qwen3.5:397b` | `all` |
| **Development/testing** | `anthropic:claude-3-haiku-20240307` | `all` |
| **Offline/self-hosted** | `ollama:qwen3.5:397b` | `all` |

## How to Run

```bash
# Full benchmark with default model
docker compose exec backend python -m scripts.benchmark --layers all none --verbose

# Compare specific models
docker compose exec backend python -m scripts.benchmark \
  --models ollama:qwen3.5:397b anthropic:claude-haiku-4-5-20251001 \
  --layers all none --verbose

# Dry run (no LLM calls)
docker compose exec backend python -m scripts.benchmark --dry-run

# Single prompt test
docker compose exec backend python -m scripts.benchmark --only slack-webhook --verbose

# Save results to custom file
docker compose exec backend python -m scripts.benchmark --output scripts/results_$(date +%Y%m%d).json
```

Results are automatically saved to `backend/scripts/benchmark_results.json`.
